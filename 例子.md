# 0. 先定一个具体场景

有 4 个参与方：

* **AS**：`ws://10.0.0.11:9000`
* **TGS**：`ws://10.0.0.11:9001`
* **GS（游戏服）**：`ws://10.0.0.12:9100`
* **客户端 A**：玩家 `linhai`
* **客户端 B**：玩家 `xiaowang`

公共固定值：

* `realm = "GAME.LOCAL"`
* `serviceName = "game/ws@10.0.0.12:9100"`

玩家 A 信息：

* `username = "linhai"`
* `password = "123456Aa!"`
* `clientId = "cli-a-001"`

玩家 B 信息：

* `username = "xiaowang"`
* `password = "654321Bb!"`
* `clientId = "cli-b-001"`

---

# 1. 先看全流程长什么样

你可以先记住这 5 句话：

1. **A 先找 AS 登录，拿到：TGT + K_c_tgs**
2. **A 再找 TGS，拿到：Service Ticket + K_c_gs**
3. **A 再找 GS，完成双向认证，拿到：sessionId**
4. **之后所有游戏业务消息，只带 `sessionId + payloadCipher + mac`**
5. **断线重连时，重新带 `ServiceTicket + 新 authenticator`，恢复旧 session**

对应流程图就是：

```text
A -> AS  : REGISTER_REQ / AS_REQ
AS -> A  : REGISTER_REP / AS_REP(TGT + K_c_tgs)

A -> TGS : TGS_REQ(TGT + authenticator)
TGS -> A : TGS_REP(ServiceTicket + K_c_gs)

A -> GS  : GS_AUTH(ServiceTicket + authenticator)
GS -> A  : GS_AUTH_OK(sessionId)

A <-> GS : 大厅 / 房间 / 战斗 / 心跳
A -> GS  : RECONNECT_REQ(断线后)
GS -> A  : RECONNECT_REP
```

```json
  "service": "AS",

  "clientId": "cli-a-001",
  "sessionId": null,
  "roomId": null,
  "status": "REQ",
  "security": {
    "enc": "RSA",
    "macAlg": "NONE",
    "keyId": "as-rsa-2026-01",
    "ticket": null,
    "authenticator": null,
    "mac": null
  },
  "payload": null,
  "payloadCipher": "Base64(RSA(...))",
  "error": null
}
```

`payloadCipher` 里面解开后的明文是：

```json
{
  "username": "linhai",
  "password": "123456Aa!",
```

AS 收到后：

1. 检查用户名是否已存在
2. 生成 `userSalt`
3. 用 `PBKDF2(password, userSalt, pwdIter)` 存密码哈希
4. 初始化这个账号的 `loginGen = 1`

```json
  "status": "REQ",
  "security": {
    "enc": "RSA",
    "macAlg": "NONE",
    "keyId": "as-rsa-2026-01",
    "ticket": null,
    "authenticator": null,
    "mac": null
  },
  "payload": null,
  "payloadCipher": "Base64(RSA(...))",
  "error": null
}
```

`payloadCipher` 解开后的明文：

```json
{
  "username": "linhai",
  "password": "123456Aa!",
  "realm": "GAME.LOCAL",
  "tgsName": "krbtgt/GAME.LOCAL",
  "nonce": "as-nonce-a-001",
  "timestampMs": 1776650410000,
  "clientId": "cli-a-001"
}
```

---

## 3.2 AS -> A：`AS_REP`

AS 验证密码成功后，会做两件事：

1. 生成 `K_c_tgs`
2. 生成一个 **TGT**，用 `K_tgs` 加密，客户端自己看不懂，只能转发给 TGS

线上发出去的是：

```json
{
  "ver": "KBGP/1.0",
  "msgType": "AS_REP",
  "service": "AS",
  "requestId": "req-as-a-001",
  "connSeq": 2,
  "timestampMs": 1776650410100,
  "clientId": "cli-a-001",
  "sessionId": null,
  "roomId": null,
  "status": "REP",
  "security": {
    "enc": "DES",
    "macAlg": "NONE",
    "keyId": null,
    "ticket": "Base64(TGT_A)",
    "authenticator": null,
    "mac": null
  },
  "payload": {
    "userId": 10001,
    "username": "linhai",
    "realm": "GAME.LOCAL",
    "userSalt": "bXlfc2FsdF9hXzAwMQ==",
    "pwdIter": 100000
  },
  "payloadCipher": "Base64(DES_with_K_user(...))",
  "error": null
}
```

其中：

* `security.ticket` 里放的是 **TGT**
* `payloadCipher` 用的是 `K_user` 加密
* `K_user = PBKDF2(password, userSalt, pwdIter, dkLen=8)`

客户端先用 `payload.userSalt + 自己输入的密码` 派生出 `K_user`，然后解 `payloadCipher`。

解开后的内容是：

```json
{
  "nonce": "as-nonce-a-001",
  "tgsName": "krbtgt/GAME.LOCAL",
  "kcTgs": "S0NUR1MtQS0wMDE=",
  "issuedAtMs": 1776650410100,
  "expireAtMs": 1776679210100,
  "renewUntilMs": 1776736810100,
  "loginGen": 17
}
```

### 这一步结束后，A 手里有什么

A 现在持有：

* `TGT_A`
* `K_c_tgs`
* `loginGen = 17`

你可以理解成：

> “AS 已经认可我是 linhai，并给了我一张去找 TGS 的临时通行证 TGT，以及我和 TGS 对话用的会话密钥 K_c_tgs。”

# 4. 玩家 A 向 TGS 申请 Game Server 的票据

这一步目标是拿到：

* **Service Ticket**
* **K_c_gs**

## 4.1 A -> TGS：`TGS_REQ`

又是一个新连接，所以 `connSeq` 再次从 `1` 开始。

```json
{
  "ver": "KBGP/1.0",
  "msgType": "TGS_REQ",
  "service": "TGS",
  "requestId": "req-tgs-a-001",
  "connSeq": 1,
  "timestampMs": 1776650500000,
  "clientId": "cli-a-001",
  "sessionId": null,
  "roomId": null,
  "status": "REQ",
  "security": {
    "enc": "DES",
    "macAlg": "HMAC-SHA256",
    "keyId": null,
    "ticket": "Base64(TGT_A)",
    "authenticator": "Base64(DES_with_K_c_tgs(...))",
    "mac": "Base64(HMAC(...))"
  },
  "payload": null,
  "payloadCipher": "Base64(DES_with_K_c_tgs(...))",
  "error": null
}
```

这里有 3 个重点：

### 1）`security.ticket`

这里放的是刚刚拿到的 `TGT_A`

### 2）`security.authenticator`

这是 A 自己临时生成的认证器，用 `K_c_tgs` 加密

解开后是：

```json
{
  "clientId": "cli-a-001",
  "timestampMs": 1776650500000,
  "nonce": "auth-tgs-a-001",
  "requestId": "req-tgs-a-001"
}
```

### 3）`payloadCipher`

这也是用 `K_c_tgs` 加密的请求体：

```json
{
  "serviceName": "game/ws@10.0.0.12:9100",
  "nonce": "tgs-nonce-a-001",
  "requestedLifeMs": 7200000
}
```

---

## 4.2 TGS 做什么

TGS 收到后：

1. 用 `K_tgs` 解开 `TGT_A`
2. 拿到：

   * `userId=10001`
   * `username=linhai`
   * `clientId=cli-a-001`
   * `K_c_tgs`
   * `loginGen=17`
3. 再用 `K_c_tgs` 解开 authenticator
4. 检查：

   * `clientId` 是否一致
   * 时间戳是否在 30 秒内
   * `loginGen` 是否还是当前值
5. 生成一个新的 **K_c_gs**
6. 生成一个 **Service Ticket**，这个票据是给 GS 看的，用 `K_gs` 加密

---

## 4.3 TGS -> A：`TGS_REP`

线上发出去：

```json
{
  "ver": "KBGP/1.0",
  "msgType": "TGS_REP",
  "service": "TGS",
  "requestId": "req-tgs-a-001",
  "connSeq": 2,
  "timestampMs": 1776650500100,
  "clientId": "cli-a-001",
  "sessionId": null,
  "roomId": null,
  "status": "REP",
  "security": {
    "enc": "DES",
    "macAlg": "HMAC-SHA256",
    "keyId": null,
    "ticket": "Base64(ST_A_FOR_GS)",
    "authenticator": null,
    "mac": "Base64(HMAC(...))"
  },
  "payload": null,
  "payloadCipher": "Base64(DES_with_K_c_tgs(...))",
  "error": null
}
```

客户端用 `K_c_tgs` 解开 `payloadCipher`，得到：

```json
{
  "nonce": "tgs-nonce-a-001",
  "serviceName": "game/ws@10.0.0.12:9100",
  "kcGs": "S0NHUy1BLTAwMQ==",
  "issuedAtMs": 1776650500100,
  "expireAtMs": 1776657700100,
  "loginGen": 17
}
```

### 这一步结束后，A 手里有什么

A 现在持有：

* `TGT_A`
* `K_c_tgs`
* `ServiceTicket_A_for_GS`
* `K_c_gs`

你可以理解成：

> “TGS 已经给我发了一张能去游戏服认证的票，以及我和游戏服会话用的密钥 K_c_gs。”

## 5.2 GS 做什么

GS 收到后：

1. 用 `K_gs` 解开 `ServiceTicket_A_for_GS`
2. 得到：

   * `userId=10001`
   * `username=linhai`
   * `clientId=cli-a-001`
   * `K_c_gs`
   * `loginGen=17`
3. 用 `K_c_gs` 解开 authenticator
4. 检查：

   * `clientId` 一致
   * 时间戳在 30 秒内
   * `loginGen` 仍有效
   * 该 authenticator 没被重放
5. 创建业务会话：

   * `sessionId = "sess-a-9001"`

---

## 5.3 GS -> A：`GS_AUTH_OK`

这是双向认证的关键。
GS 要证明“我确实拿到了 `K_c_gs`，不是假服务器”。

线上发出去：

```json
{
  "ver": "KBGP/1.0",
  "msgType": "GS_AUTH_OK",
  "service": "GS",
  "requestId": "req-gs-auth-a-001",
  "connSeq": 2,
  "timestampMs": 1776650510020,
  "clientId": "cli-a-001",
  "sessionId": "sess-a-9001",
  "roomId": null,
  "status": "REP",
  "security": {
    "enc": "DES",
    "macAlg": "HMAC-SHA256",
    "keyId": null,
    "ticket": null,
    "authenticator": null,
    "mac": "Base64(HMAC(...))"
  },
  "payload": null,
  "payloadCipher": "Base64(DES_with_K_c_gs(...))",
  "error": null
}
```

解开 `payloadCipher` 后：

```json
{
  "clientTimestampMs": 1776650510000,
  "nonce": "auth-gs-a-001",
  "serverTimeMs": 1776650510020,
  "sessionId": "sess-a-9001",
  "serviceTicketExpireAtMs": 1776657700100,
  "heartbeatIntervalMs": 10000,
  "reconnectGraceMs": 30000
}
```

客户端只要检查：

* `clientTimestampMs` 是否等于自己刚发的时间
* `nonce` 是否等于自己刚发的 nonce

就知道：

> “对方确实能解我的票据，也确实拿到了 K_c_gs，它就是合法 GS。”

---

# 6. 到这里为止，认证阶段结束了

A 现在手里最重要的 3 个东西是：

* `ServiceTicket_A_for_GS`
* `K_c_gs`
* `sessionId = "sess-a-9001"`

### 从现在开始，业务消息怎么发？

**不再带 ticket，不再带 authenticator。**

因为 A 和 GS 已经共享 `K_c_gs` 了，后面游戏消息只需要：

* `sessionId`
* `payloadCipher`
* `mac`

这点特别重要。

# 11. 断线重连怎么走

这一步是很多人第一次做最容易绕的地方。

假设 A 在对战中突然断网了。

---

## 11.1 GS 发现 A 掉线

GS 会把 A 的会话标成：

* `sessionId = "sess-a-9001"`
* `status = DISCONNECTED_GRACE`

并且开始计时：

* `reconnectGraceMs = 30000`

这 30 秒里：

* A 还能回来
* GS 继续用“空输入”推进比赛

---

## 11.2 A 重新连上 GS

这是**一条新的 WebSocket 连接**，所以：

* `connSeq` 又从 `1` 开始

但这不是新登录，所以：

* `sessionId` 还是旧的：`sess-a-9001`

---

## 11.3 A -> GS：`RECONNECT_REQ`

```json
{
  "ver": "KBGP/1.0",
  "msgType": "RECONNECT_REQ",
  "service": "GS",
  "requestId": "req-reconnect-a-001",
  "connSeq": 1,
  "timestampMs": 1776650535000,
  "clientId": "cli-a-001",
  "sessionId": "sess-a-9001",
  "roomId": "room-123",
  "status": "REQ",
  "security": {
    "enc": "DES",
    "macAlg": "HMAC-SHA256",
    "keyId": null,
    "ticket": "Base64(ST_A_FOR_GS)",
    "authenticator": "Base64(DES_with_K_c_gs(...))",
    "mac": "Base64(HMAC(...))"
  },
  "payload": null,
  "payloadCipher": "Base64(DES_with_K_c_gs(...))",
  "error": null
}
```

`authenticator` 解开后：

```json
{
  "clientId": "cli-a-001",
  "sessionId": "sess-a-9001",
  "timestampMs": 1776650535000,
  "nonce": "auth-reconnect-a-001",
  "requestId": "req-reconnect-a-001"
}
```

`payloadCipher` 解开后：

```json
{
  "lastSeenServerConnSeq": 95,
  "lastProcessedSeq": 108
}
```

### 为什么重连还要带 `ServiceTicket`

因为 GS 要再次确认：

* 你还是那个合法用户
* 你手里还持有合法票据
* 票据还没过期
* `loginGen` 没变

### 为什么 authenticator 必须重新生成

因为旧 authenticator 不允许重放。
重连本质上是一次新的认证动作，所以要带一个新的 authenticator。

# 13. 最后再看一次：每个阶段客户端到底多了什么

这个很重要，我给你总结成一张“小抄”。

## 注册后

客户端知道：

* `userId`
* `userSalt`
* `pwdIter`

---

## `AS_REP` 后

客户端新增：

* `TGT`
* `K_c_tgs`
* `loginGen`

---

## `TGS_REP` 后

客户端新增：

* `ServiceTicket`
* `K_c_gs`

---

## `GS_AUTH_OK` 后

客户端新增：

* `sessionId`

---

## 进入业务阶段后

客户端发消息只需要：

* `sessionId`
* `payloadCipher`
* `mac`

---

## 断线重连时

客户端必须本地缓存好：

* `ServiceTicket`
* `K_c_gs`
* `sessionId`
* 最近一次 `lastProcessedSeq`

不然就没法恢复。

# 14. 你最容易混淆的 4 个点，我直接替你拆开

## 1）`TGT` 和 `ServiceTicket` 的区别

* `TGT`：给 **TGS** 看的
* `ServiceTicket`：给 **GS** 看的

---

## 2）为什么认证完了以后不再发 `ticket`

因为已经完成 `GS_AUTH` 了，GS 已经给你建好了 `sessionId`，双方也已经共享 `K_c_gs`。
后面继续发 `ticket` 只是浪费。

---

## 3）`connSeq` 和 `payload.seq` 为什么要分开

* `connSeq`：保证这条 WebSocket 连接上的报文顺序
* `payload.seq`：保证对战输入序号连续

比如你断线重连后：

* `connSeq` 会重新从 `1` 开始
* 但输入序号 `payload.seq` 应该继续从 `109`、`110` 往后走

---

## 4）为什么重连还要带 authenticator

因为重连不是“续传 TCP 包”，而是“新连接上的重新认证”。
你要再次证明“我是原来的那个合法客户端”。
