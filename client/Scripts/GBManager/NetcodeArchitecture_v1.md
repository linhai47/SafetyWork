# 联机架构方案 v1

## 目标

先做出这个游戏的第一版可玩联机对战。

这一版只做最核心的 PvP 内容：

- 2 个玩家进房间
- 出生
- 左右移动
- 跳跃 / 下落
- 朝向
- 1 把近战武器
- 1 把远程武器
- 命中
- 击退 / 击飞
- 死亡 / 出界
- 复活或扣命

这一版先不做：

- 任务
- NPC
- 商店
- 副本
- 存档
- 对话系统
- roguelike 元进度
- 背包同步

这份方案的核心思想只有一句话：

- 同步输入，不直接同步位置
- 服务器负责最终判定
- 本地玩家做预测
- 远端玩家用快照表现

---

## 先说结论

你现在这个项目，不是“在一个新项目里加联机”。

而是：

- 在一个旧的单机 ARPG / RPG 项目基础上
- 抽出里面最好用的战斗部分
- 重组成一个 2D 平台对战游戏
- 再在这个战斗核心上做联机

所以现在最大的工作，不是先接网络 SDK。

而是先把下面这些东西分开：

- 哪些是对战核心
- 哪些是旧项目包袱
- 哪些脚本以后必须能跑在联机模拟里
- 哪些脚本暂时不要参与对战

---

## 这类游戏该用什么联机思路

你这个项目像：

- 任天堂大乱斗
- 2D 平台格斗 / 对战
- 带枪械和刀剑

这种游戏最怕的不是“人能不能动”，而是：

- 近战命中判定不一致
- 击飞不同步
- 子弹位置不同步
- 双方都觉得自己打中了
- 一边已经死了，另一边还活着

所以这类游戏第一版最稳的方式是：

1. 客户端每一帧 / tick 发输入
2. 本地先立即预测自己的移动和操作
3. 服务器用同样输入跑真正的权威结果
4. 服务器把结果打成快照发回来
5. 本地如果预测错了，就回滚 / 校正
6. 远端玩家只负责平滑显示，不自己决定结果

不要一开始就做这种：

- 直接同步 `transform.position`
- 双方本地各算各的攻击命中
- 子弹谁先看到谁说了算

这种做法刚开始看起来省事，但一到：

- 近战
- 击退
- 挡反
- 平台边缘
- 同时出手

就会开始炸。

## 推荐你拆成 4 层

后面整个联机对战，建议你按 4 层理解。

## 1. Match 层：比赛层

负责：

- 房间状态
- 玩家加入 / 离开
- 开场倒计时
- stock / 命数
- 回合结束
- 复活
- 游戏结束

你现在最接近这一层的脚本是：

- `GameManager.cs`

后面它可以慢慢演化成：

- `VersusMatchManager`

重要原则：

- 比赛层不要直接读键盘鼠标
- 比赛层只负责“比赛规则”

---

## 2. Simulation 层：战斗模拟层

负责：

- 移动
- 跳跃
- 重力
- 翻面
- 攻击前摇 / 出手 / 后摇
- 子弹生成
- 命中判定
- 伤害
- 击退 / 击飞
- 死亡

你现在最接近这一层的是：

- `Entity.cs`
- `Player.cs`
- `PlayerStates/`
- `Weapon/`
- `Objects/Projectile.cs`

重要原则：

- 这一层必须只吃“输入命令”或“权威状态”
- 不应该依赖 UI、任务、商店、存档、对话系统

---

## 3. Netcode 层：网络层

负责：

- 输入命令打包
- tick 管理
- 客户端预测
- 服务器校正
- 快照缓存
- 远端插值

你现在已经有一点雏形：

- `GBManager/网络相关/PlayerInputCmd.cs`
- `GBManager/网络相关/MatchSnapshot.cs`
- `GBManager/网络相关/InputPacker.cs`
- `GBManager/网络相关/ClientReceiver.cs`
- `GBManager/网络相关/MockNetwork.cs`

重要原则：

- 网络层只负责传输和应用状态
- 不要让网络层自己决定游戏规则

---

## 4. Presentation 层：表现层

负责：

- 动画
- 特效
- 摄像机
- UI
- 音效
- hit stop
- 屏幕震动

对应你现在项目里的：

- `VFX/`
- `Camera/`
- `UI/`
- `Systems/HitStopController.cs`

重要原则：

- 表现层可以稍微慢半拍
- 但真正的结果必须来自权威模拟

## 联机里你需要的 4 种运行身份

同样一个“玩家角色”，在联机里其实可能有 4 种不同身份。

## 1. 本地预测玩家

就是“我自己操作的角色”。

它要做的事：

- 读取本地输入
- 立即响应，让手感不延迟
- 把输入发给服务器
- 收到服务器结果后进行校正

它应该拥有：

- 输入采集
- 本地预测
- 输入缓存
- 校正逻辑

它不应该拥有：

- 最终伤害结算权

---

## 2. 远端代理玩家

就是“别人控制的那个角色，在我屏幕上显示出来的版本”。

它要做的事：

- 不读本地输入
- 不做本地最终判定
- 只根据服务器快照平滑显示

它应该拥有：

- 快照缓存
- 插值表现
- 动画同步

它不应该拥有：

- 本地键鼠输入
- 最终命中结果判定

---

## 3. 服务器权威玩家

就是服务器 / 房主那边真正负责算结果的那份角色。

它要做的事：

- 按 tick 顺序消费输入
- 跑真正的移动和战斗模拟
- 生成权威状态

它必须决定：

- 角色移动结果
- 攻击是否命中
- 子弹什么时候生成
- 伤害是多少
- 击飞多少
- 死亡
- 复活

---

## 4. 共享表现对象

比如：

- UI
- 特效
- 摄像机

这些可以根据状态做表现，但不要掌握“最终真相”。

## 你现在的数据结构不够，后面应该怎么扩

## 1. 输入命令 PlayerInputCmd

你现在的 `PlayerInputCmd` 只有：

- `seq`
- `moveX`
- `moveY`

这对平台对战来说太少了。

后面建议至少扩成这样：

```csharp
[Serializable]
public struct PlayerInputCmd
{
    public int tick;
    public float moveX;
    public bool jumpPressed;
    public bool jumpHeld;
    public bool attackPressed;
    public bool attackHeld;
    public bool skillPressed;
    public float aimX;
    public float aimY;
    public sbyte facingHint;
}
```

为什么这么改：

- 平台游戏里 `moveY` 往往不如“跳跃按下 / 长按”重要
- 枪械和刀都需要更明确的攻击输入
- 远程武器需要瞄准方向

---

## 2. 玩家权威状态 PlayerStateSnapshot

你需要一份“玩家在某个 tick 的权威状态”。

例如：

```csharp
[Serializable]
public struct PlayerStateSnapshot
{
    public int tick;
    public int playerId;
    public float posX;
    public float posY;
    public float velX;
    public float velY;
    public sbyte facing;
    public byte locomotionState;
    public byte combatState;
    public int hp;
    public int stocks;
    public bool grounded;
}
```

它的作用是：

- 给本地玩家做校正
- 给远端玩家做平滑播放
- 给比赛层判断死亡 / 复活 / 命数

---

## 3. 比赛快照 MatchSnapshot

你现在的 `MatchSnapshot` 只有一个位置，肯定不够。

后面应该更像这样：

```csharp
[Serializable]
public struct MatchSnapshot
{
    public int tick;
    public PlayerStateSnapshot player1;
    public PlayerStateSnapshot player2;
}
```

再往后你还可以继续加：

- 子弹列表
- 新生成的攻击事件
- 击杀事件
- 复活事件

但第一版先别做太复杂。

## tick 怎么跑

建议你先用固定 tick。

例如：

- 30 tick / 秒

可以先基于 Unity 的 `FixedUpdate` 来做第一版。

每个 tick 做这些事：

1. 读取输入
2. 打上 tick 编号
3. 发给服务器
4. 本地立刻预测
5. 服务器用同样输入跑真正模拟
6. 服务器广播权威快照
7. 本地收到后校正，远端收到后插值显示

最重要的一点：

- 本地预测和服务器模拟，尽量走同一套代码

如果客户端和服务器的角色移动 / 攻击逻辑不是同一套，后面校正会抖得很严重。

## 战斗里的权责划分

你这个游戏里，这几条最好定死，不然后面很容易乱。

## 移动

- 本地先预测
- 服务器最终确认

## 朝向

- 可以由输入或瞄准方向推导
- 最终以服务器为准

## 近战命中

- 由服务器决定有没有打中

## 子弹生成

- 客户端可以先播枪口火花
- 但真正的子弹生成由服务器决定

## 伤害和击飞

- 只能服务器算

## 死亡和复活

- 只能服务器决定

这个规则对你非常重要，因为你游戏里有：

- 枪
- 刀
- 挡反
- 击飞
- 平台边界

这些一旦双方各算各的，很快就会不同步。

## 你最容易踩的坑

一个最常见的错误思路是：

- “远端玩家就还是用同一个 `Player` 预制体，只是把输入关掉不就好了？”

这个思路短期看很方便，长期非常容易爆炸。

因为你现在这个 `Player` 还带着很多单机包袱：

- 本地输入假设
- UI 假设
- 本地状态机时序
- 动画事件直接推动战斗
- 旧项目依赖

所以更推荐的是：

- 本地玩家 = 预测模拟
- 远端玩家 = 快照代理
- 服务器玩家 = 权威模拟

外观可以是同一个 prefab。

但运行职责不能完全一样。

如果从这份方案继续往下走，我最建议的下一步是：

1. 增加一个网络角色枚举
2. 新建 `PlayerNetController`
3. 扩展 `PlayerInputCmd`
4. 让 `PlayerCombatController` 改为消费命令，而不是自己读输入
5. 把 `Player.cs` 里对战无关的旧系统依赖先隔离出去

这一步做完以后，你的项目才算真正进入：

- “可联机化的战斗架构”

而不是：

- “单机项目 + 一些网络实验脚本”

## 最后一句判断

你这个项目现在已经够资格开始做联机了。

但前提是你要把它当成：

- 一个以战斗为核心的联机对战项目

而不是继续把它当成：

- 一个老 RPG 项目顺手加联机功能

对你这个类型的游戏来说，联机必须尽早站到战斗核心旁边，而不是挂在旧的任务 / UI / 存档体系外面。
